# zkCompression en Solana: Ejemplos en Rust Nativo sin Anchor

Entiendo que quieres ver cómo se implementan cuentas y estados comprimidos "normales" (es decir, usando el SDK de Light Protocol en Rust puro, sin el framework Anchor). En Solana, las cuentas comprimidas no se almacenan directamente en la cadena; en su lugar, sus datos se mantienen off-chain y solo se almacenan hashes en árboles Merkle on-chain. Esto reduce costos drásticamente, pero las actualizaciones siguen un modelo UTXO (Unspent Transaction Output): consumes el hash viejo y produces uno nuevo.

Usando `light-sdk`, el código maneja esto a través de wrappers como `LightAccount` para inicializar, mutar y hacer CPI (Cross-Program Invocation) al Light System Program. Los estados se definen en structs que derivan `LightDiscriminator`, `BorshSerialize` y `BorshDeserialize` para hashing y serialización.

A continuación, un guía quickstart basado en ejemplos oficiales de Light Protocol. Para un programa simple como un "contador" o "mensaje", el código se ve así. (Adaptado de sus docs y ejemplos en GitHub).

## Dependencias en `Cargo.toml`
Para un programa nativo:
```toml
[package]
name = "my_program"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib", "lib"]

[dependencies]
borsh = "1.5"  # Para serialización
light-sdk = "0.16.0"  # SDK para cuentas comprimidas
light-macros = "0.16.0"  # Para macros como pubkey!
solana-program = "2.2"  # SDK base de Solana
```

## Estructura de la Cuenta Comprimida
Define tu estado como un struct serializable. Usa `LightDiscriminator` para identificación de tipo:
```rust
use borsh::{BorshDeserialize, BorshSerialize};
use light_sdk::LightDiscriminator;
use solana_program::pubkey::Pubkey;

#[derive(Debug, Default, Clone, BorshSerialize, BorshDeserialize, LightDiscriminator)]
pub struct MyCompressedAccount {
    pub owner: Pubkey,  // Dueño de la cuenta
    pub counter: u64,   // Estado: un contador simple (o mensaje: String)
}
```
- **Cómo se ve el estado**: Off-chain, es solo datos en el struct. On-chain, es un hash en el State Merkle Tree. No hay "cuenta" tradicional con lamports o data overwritable.

## Punto de Entrada y Procesamiento de Instrucciones
En `src/lib.rs`, define el entrypoint y maneja discriminadores manualmente (sin Anchor's `#[program]`):
```rust
#![allow(unexpected_cfgs)]
use borsh::{BorshDeserialize, BorshSerialize};
use light_macros::pubkey;
use light_sdk::{
    account::sha::LightAccount,
    address::v1::derive_address,
    cpi::{
        v1::{CpiAccounts, LightSystemProgramCpi},
        CpiSigner,
        InvokeLightSystemProgram,
    },
    derive_light_cpi_signer,
    error::LightSdkError,
    instruction::{PackedAddressTreeInfo, ValidityProof, account_meta::CompressedAccountMeta},
    LightDiscriminator,
};
use solana_program::{
    account_info::AccountInfo, entrypoint, program_error::ProgramError, pubkey::Pubkey,
};

pub const ID: Pubkey = pubkey!("TuProgramIDAqui");  // Reemplaza con tu program ID
pub const LIGHT_CPI_SIGNER: CpiSigner = derive_light_cpi_signer!("TuProgramIDAqui");

entrypoint!(process_instruction);

#[repr(u8)]
#[derive(Debug)]
pub enum InstructionType {
    Create = 0,
    Update = 1,  // O Increment para un contador
}

impl TryFrom<u8> for InstructionType {
    type Error = LightSdkError;
    fn try_from(value: u8) -> Result<Self, Self::Error> {
        match value {
            0 => Ok(InstructionType::Create),
            1 => Ok(InstructionType::Update),
            _ => Err(LightSdkError::InvalidInstructionDiscriminator),
        }
    }
}

#[derive(BorshSerialize, BorshDeserialize)]
pub struct CreateInstructionData {
    pub proof: ValidityProof,                // Prueba ZK del cliente
    pub address_tree_info: PackedAddressTreeInfo,  // Info del árbol de direcciones
    pub output_state_tree_index: u8,         // Índice de salida en el state tree
}

#[derive(BorshSerialize, BorshDeserialize)]
pub struct UpdateInstructionData {
    pub proof: ValidityProof,                // Prueba ZK
    pub account_meta: CompressedAccountMeta, // Metadata de la cuenta existente (árbol, índice, etc.)
    pub current_counter: u64,                // Estado actual para hashing
}

pub fn process_instruction(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    instruction_data: &[u8],
) -> Result<(), ProgramError> {
    if program_id != &ID {
        return Err(ProgramError::IncorrectProgramId);
    }
    if instruction_data.is_empty() {
        return Err(ProgramError::InvalidInstructionData);
    }

    let discriminator = InstructionType::try_from(instruction_data[0])
        .map_err(|_| ProgramError::InvalidInstructionData)?;
    
    match discriminator {
        InstructionType::Create => {
            let data = CreateInstructionData::try_from_slice(&instruction_data[1..])
                .map_err(|_| ProgramError::InvalidInstructionData)?;
            create(accounts, data)
        }
        InstructionType::Update => {
            let data = UpdateInstructionData::try_from_slice(&instruction_data[1..])
                .map_err(|_| ProgramError::InvalidInstructionData)?;
            update(accounts, data)
        }
    }
}
```

## Instrucción: Crear Cuenta Comprimida
Crea una nueva cuenta con dirección derivada (como PDA, pero comprimida):
```rust
pub fn create(
    accounts: &[AccountInfo],
    instruction_data: CreateInstructionData,
) -> Result<(), ProgramError> {
    let signer = accounts.first().ok_or(ProgramError::NotEnoughAccountKeys)?;
    let light_cpi_accounts = CpiAccounts::new(signer, &accounts[1..], LIGHT_CPI_SIGNER);

    // Deriva dirección persistente (semilla: "counter" + signer)
    let (address, address_seed) = derive_address(
        &[b"counter", signer.key.as_ref()],
        &instruction_data.address_tree_info.get_tree_pubkey(&light_cpi_accounts)
            .map_err(|_| ProgramError::NotEnoughAccountKeys)?,
        &ID,
    );

    let new_address_params = instruction_data.address_tree_info
        .into_new_address_params_packed(address_seed);

    // Inicializa la cuenta comprimida
    let mut my_compressed_account = LightAccount::<MyCompressedAccount>::new_init(
        &ID,
        Some(address),
        instruction_data.output_state_tree_index,
    );
    my_compressed_account.owner = *signer.key;
    my_compressed_account.counter = 0;  // Estado inicial

    // CPI al Light System Program para insertar hash en árboles
    LightSystemProgramCpi::new_cpi(LIGHT_CPI_SIGNER, instruction_data.proof)
        .with_light_account(my_compressed_account)?
        .with_new_addresses(&[new_address_params])
        .invoke(light_cpi_accounts)?;
    
    Ok(())
}
```
- **Notas**: El cliente proporciona `proof` y `address_tree_info`. La cuenta se inicializa con `LightAccount::new_init`, que maneja el hashing.

## Instrucción: Actualizar Estado Comprimido (ej. Incrementar Contador)
Actualiza siguiendo UTXO: muta el struct, hashea nuevo estado:
```rust
pub fn update(
    accounts: &[AccountInfo],
    instruction_data: UpdateInstructionData,
) -> Result<(), ProgramError> {
    let signer = accounts.first().ok_or(ProgramError::NotEnoughAccountKeys)?;
    let light_cpi_accounts = CpiAccounts::new(signer, &accounts[1..], LIGHT_CPI_SIGNER);

    // Wrapper mutable para la cuenta existente
    let mut my_compressed_account = LightAccount::<MyCompressedAccount>::new_mut(
        &ID,
        &instruction_data.account_meta,
        MyCompressedAccount {
            owner: *signer.key,
            counter: instruction_data.current_counter,  // Estado actual del cliente
        },
    )?;

    // Actualiza el estado
    my_compressed_account.counter += 1;  // O cambia a nuevo valor

    // CPI: Nullifica viejo hash, inserta nuevo
    LightSystemProgramCpi::new_cpi(LIGHT_CPI_SIGNER, instruction_data.proof)
        .with_light_account(my_compressed_account)?
        .invoke(light_cpi_accounts)?;
    
    Ok(())
}
```
- **Notas**: `account_meta` incluye info del árbol y output index. El cliente fetcha `proof` con `getValidityProof()`. No se "lee" on-chain; se verifica via ZK.

## Cómo se Ven las Cuentas y Estados
- **Sin Anchor**: Manejas todo manualmente: deserialización con Borsh, validación de accounts, discriminadores. Es más verboso, pero da control total.
- **Estados Comprimidos**: Son structs off-chain. On-chain: solo hashes en Merkle trees (via Account Compression Program). Usa exploradores como Solana Explorer con soporte para compressed accounts para ver hashes.
- **Testing**: Usa `light test-validator` local. Cliente en TS con `@lightprotocol/stateless.js` para fetch proofs y metas.

Para ejemplos completos como un contador, mira el repo en GitHub: [counter/native](https://github.com/Lightprotocol/program-examples/tree/main/counter/native). Si necesitas ajustes para tu caso (ej. en español o más detalles), ¡dime!
